## FAQ 页面 — 需要你亲自核实的数字

这些是我在 FAQ 里看到的具体数字声明,如果产品还没正式启动,这些大概率是占位文案。请逐条确认是真实数据还是需要删除/改写:

```
□ "每月处理 5000 万+ 次生产环境请求"
□ "标准 SLA 为 99.9%"
□ "在 180 多个国家运营"
□ "在北京和香港部署了边缘节点"
□ "Discord（即将上线）"
□ "除非你主动请求,你的数据不会离开中国"（这条涉及合规承诺，务必确认真实性）
```

**为什么这个重要:** 你的核心定位是"零加价、官方原生模型、透明代理"——这个定位建立在"诚实"这个信任基础上。如果用户验证发现具体数字是编的（比如查不到 5000 万请求的任何证据），会反过来怀疑"零加价"是不是也是营销话术。建议：

- 真实存在的就留着
- 还没达到的，删掉具体数字，换成定性描述。比如"99.9% SLA"如果还没有正式 SLA 承诺，改成"我们以高可用性为设计目标"
- "数据不会离开中国"这种合规性表述如果不确定，建议直接删除或换成"详情见隐私政策"，避免法律风险

## 整体信息架构建议

现在的文档导航顺序：
```
快速开始 → API 参考 → 认证 → 代码示例 → 常见问题
```

建议改为：
```
工作原理（新增，放在最前）→ 快速开始 → API 参考 → 认证 → 代码示例 → 常见问题
```

理由：第一性原理的核心就是「先建立正确的心智模型，再讲操作步骤」。现在用户一上来就看到操作命令，却始终不知道自己在操作什么、数据流向哪里——这是大多数 API 网关文档的通病，也是建立信任最薄弱的一环。把"工作原理"放在最前面，等于在用户跑第一个请求之前，先回答了他心里那个没问出口的疑虑："我把 Key 和请求都给你了，你们到底拿去干嘛了？"
